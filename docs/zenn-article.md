# はじめに：私が直面した現場の課題

会社でClaudeやCopilotの導入を推進を担当してるんですが、導入したはいいものの、利用率は少ない状態でした。また、使い方がわからない人は、全く利用はしないんですが、使いこなしてる人は、どんどん業務をAIに置き換えていくという、「AI活用格差」を実感していました。

使い方のレクチャーしたり、いかに業務を効率化できるかということを伝えても、AI活用格差は広がるばかりでした。

この課題を解決したいのと、業務に対するAIの活用方法を、暗黙知として考え、その暗黙知を形式知にして、会社に浸透させたいと思い「コダマ」を考えました。

今回のハッカソンを契機に、AIを使いこなすという暗黙知を社内に浸透させるエージェントを作りたいと思いました。  
その想いで、Microsoft Agent Hackathon 2026に向けて開発したのが 『Kodama』 です。

# コンセプト：まっさらなAIより、みんなが育てたAIを。

「自由にプロンプトを書いてください」

と、まっさらなAIを渡されるのは、実は多くの人にとって心理的ハードルが高いお願いなんじゃないでしょうか。

そこでKodamaが目指すのは、「みんなが育てたAI」です。

オフィスの給湯室で交わされる「それ、こうやったらすぐ終わったよ」という何気ない立ち話。
あの自然な知見の共有に、システムでレバレッジをかける。

ゼロからAIの使い方を考えるのではなく、先輩や同僚が試行錯誤して見つけたコツを、ただなぞってみるだけでいい。
AI活用の最初の一歩をそっと後押しする。それがKodamaです。

# Kodamaが提供する体験（デモシナリオ）

Kodamaは**「困っている人と成功事例を、限りなくシームレスにマッチングする」**AI活用提案エージェントです。

一般的なチャットボットとは、設計の思想が少し違います。「聞かれたら答える」のではなく、「その人の今の業務に刺さるものを、選ぶだけで使えるように提示する」ことを目指しました。

## 社員視点：今日の成功事例を眺める

Kodamaを開くと、まず**「今日のおすすめ」**が目に飛び込んできます。



これは日付をシードにしたアルゴリズムで選ばれるので、同じ日にアクセスした社員はみんな同じ事例を見ることになります。「あ、今日トップに出てたあのプロンプト、使えそうだったね」という共通認識を生むためのちょっとした工夫です。

続けて画面に並ぶのは**業務カテゴリのグリッド**です。
「月次レポート作成」や「議事録要約」といったカテゴリを選ぶと、そこに紐づく成功事例がカード形式で並びます。



各カードには、こんな情報がまとまっています。

- 誰の事例か（例：「営業部 佐藤さん」）
- 何が変わったか（例：「月8h → 2.5h（約70%削減）」）
- 今すぐ使えるプロンプト（ワンクリックでコピー可能）



事例の下には「役立ちましたか？」というシンプルなフィードバックボタン（👍 / 🤔）だけを置きました。「いいね数」のランキングはあえて作っていません。競争や監視のプレッシャーを与えず、ただ自分のために「最近試したもの」として履歴（localStorage）に残るだけの、静かな設計にこだわりました。

---

ここで「なぜチャットUIにしなかったのか」について触れておきます。

実は、Kodamaの最初のプロトタイプはChainlitを使ったチャットUIでした。でも、テストしてみると致命的な問題にぶつかりました。**AIに不慣れな人ほど、「チャットの窓に何を入力していいか分からない」**のです。

私たちが一番助けたいのはその層です。だから思い切ってチャットを捨て、ダッシュボード型に振り切りました。ぽちぽち選んで、気に入ったらコピーする。ゼロからのテキスト入力は一切不要にしています。

## DX推進担当者視点：誰に何を届けるか

管理者モードでは、支援が必要そうな社員のリストと、「推薦事例 + 次のアクション（戦略）」が一覧で提示されます。



ユーザーを選ぶと、Agentが**その人の業務文脈をクエリにしてセマンティック検索**を走り、関連度の高い成功事例を探してきます。（「自分の事例を自分に勧められる」というマヌケな挙動を防ぐため、本人の事例は自動で除外されます）。

事例が見つかると、Agentは2つの戦略を提案してくれます。

**戦略A：本人への紹介**

> 「田中さん、月次レポートでお困りのようですね。以前、佐藤さんが似た業務を効率化されていたので、15分ほどお時間とってノウハウを聞いてみませんか？」
> （※プレビューをその場で確認可能）

**戦略B：テンプレとして共有**

> 「田中さん、月次レポート作成の効率化に、社内で使われているプロンプトのテンプレートを共有します。」
> （※相手の業務名に合わせてローカライズされたプロンプトが生成される）



どちらもワンクリックでメッセージが生成されます。人間（DX担当者）は「この人にはどちらのアプローチが良さそうか」を判断するだけで済みます。

ちなみに、Agentが提案する事例は**最大3件**までと固く制限しています。選択肢が多すぎると人は選べなくなります。「どれが一番効果的か」を絞り込むこと自体も、Agentの重要な仕事です。

# 裏側で働く5つのAgentたち（技術構成）

Kodamaのバックエンドは、Azure AI Foundry Agent Service上で動く**5つのAgentの協調**によって成り立っています。

## アーキテクチャ全体像

```
[Webダッシュボード (React + Vite + TypeScript)]
         ↕ REST API
[FastAPI バックエンド]
         ↕ Azure AI Foundry SDK
[Orchestrator Agent]
  ├─ Observer Agent   ← Microsoft Graph で観測
  ├─ Collector Agent  ← 困りごとの構造化・本人確認
  ├─ Matcher Agent    ← セマンティック検索・戦略選択
  └─ Proposer Agent   ← 個別提案の生成
         ↕
[Azure OpenAI (GPT-4o 系)]
[Cosmos DB / Azure AI Search]
```



## Orchestrator Agent：全体の指揮官

OrchestratorはユーザーとKodamaの**唯一の窓口**です。ユーザーの入力を解釈し、function toolと4つの子Agentを適切に呼び分けます。

Instructionsには判断パターンが明示されています：

- **観測情報の引用**（「田中さんがこう投稿していた」）→ まず`tool_fetch_signals`で補完 → `tool_semantic_search`で関連事例を検索 → 提案
- **能動的な検索依頼**（「高橋さんに合う事例を探して」）→ `exclude_user_id`を設定して`tool_semantic_search`を呼ぶ
- **一般的な質問・雑談** → tool呼び出し不要で通常回答

「最大3件」「戦略A/Bを必ず両方提示」「内部IDを絶対表示しない」「本人の事例を本人に推薦しない」——こうした厳守ルールがInstructionsに直接書かれています。LLMがルールを「覚える」ではなく、**Instructionsで強制する**設計です。

## Observer Agent：組織活動の感覚器

ObserverはMicrosoft Graph APIを通じて、オプトイン済み社員の活動から困りごとのシグナルを検知します。

検知対象は3種類：

1. 「これどうやって」「分からない」といった質問的メッセージ（Teams投稿）
2. Copilotを使い始めて途中で止めた痕跡（セッションログ）
3. 毎月同じ手作業を繰り返しているパターン（Calendarログ）

出力はJSON構造化データで、`source_signal`（`teams_message` / `copilot_abandon` / `repetitive_pattern`）・`raw_excerpt`・`business_context_hint`の3フィールドが含まれます。**Collector Agentへの渡し役**に徹し、永続化は一切行いません。

MVPでは実データ未準備時にモックを返すフォールバックも用意しています。

## Collector Agent：Human-in-the-Loopの守り人

Collectorの仕事は**「データを保存してよいかどうかを確かめること」**です。

Observerから渡された候補シグナルを構造化し、まず本人確認のメッセージを生成します。

> 「月次レポート作業でお困りですか？」

本人が「はい」と回答した場合のみ、`save_pain_point` toolを呼んでCosmos DBに保存します。「困っていない」と回答した場合は永続化せず、同種シグナルを7日間抑制します。

**PII（個人情報）はuser_id（内部ID）のみ扱い、氏名・メールアドレスはDBに保存しません。** 表示時にのみMicrosoft Graphから取得する設計です。

## Matcher Agent：決定論とLLMのハイブリッド

Matcherは「誰と誰を繋ぐか」の判断を担います。設計のポイントは**決定論でスコアを計算し、LLMで最終戦略を選ぶ**ハイブリッド構成です。

スコアリング式：

```python
final_score = (cosine_similarity × 0.7) + (reproducibility_score × 0.3) + business_type_bonus(0.2)
```

- **コサイン類似度（重み0.7）**：困りごとテキストをtext-embedding-3-small（1536次元）でベクトル化し、成功事例との類似度を計算
- **再現可能性スコア（重み0.3）**：その事例が他者でも再現できる可能性（0〜1）。成功者のスキル依存度の逆数として登録
- **業務種別ボーナス（+0.2）**：クエリテキストに成功事例の`business_type`が含まれている場合に加算

embeddingが未登録の場合は`business_type`のテキストマッチへ自動フォールバックします。本番（Azure AI Search）への切り替えも関数単位で差し替え可能な抽象化になっています。

成功事例が0件だった場合（Cold Start）は、業務カテゴリに合うテンプレートプロンプトを提示し、「最初の成功事例を作ってください」と促します。

## Proposer Agent：文脈に合わせた提案の生成

ProposerはMatcherが選んだ成功事例を元に、依頼者向けのメッセージをカスタマイズします。

**戦略A（紹介）の場合：**

```
{成功者名} さま

お忙しい中失礼します。{対象者名} さまが {業務名} の効率化で困っており、
以前 {成功者名} さまが実施された取り組みが参考になりそうです。
15分ほどお時間いただき、ご経験を共有いただけませんでしょうか？
```

**戦略B（テンプレ共有）の場合：**
成功者が使ったプロンプトを、対象者の業務名にローカライズして生成します。「月次レポート作成向けプロンプト」をそのまま渡すのではなく、対象者の業務文脈に合わせた形に書き換えます。

Instructionsには「出典（成功者名）を必ず示す」「押し付けがましい表現を避け、選択肢として提示する」という制約が書かれています。

## 実装上のこだわりポイント

**① フィードバックをサーバーに送らない（MVP段階）**

「役立ちましたか？」ボタンのフィードバックは、今はブラウザのlocalStorageに留めています。サーバーに集計データを送る仕組みは意図的に後回しにしました。最初から立派な集計基盤を作るより、まずは「プレッシャーなく使われる体験」を作ることを優先したかったからです。

**② embeddingはlazy loadで、失敗してもサービスを止めない**

FastAPIの起動時（lifespan）に成功事例のembeddingを一括登録します。Azure OpenAIの呼び出しが失敗した場合は、embeddingなしで全件再ロードしてサービスを継続します。検索もベクトル検索からテキストマッチへ透過的にフォールバックするように組みました。

**③ ConnectedAgentToolで子Agentを「必要なときだけ」呼ぶ**

Orchestratorは4つの子Agentを保有していますが、「基本はfunction toolで完結させ、判断が難しいときのみ子Agentに委譲する」方針をとっています。LLMのコール数（コストとレイテンシ）を抑えつつ、複雑なケースへの対応力を残すバランスを狙いました。

**④ DefaultAzureCredentialで認証コードを変えない**

ローカルではaz loginの認証情報、Azure Container AppsではManaged Identityを自動で使い分けます。環境ごとに認証コードを書き替える手間がなくなり、シークレットを環境変数に晒す必要もありません。

# 今後の展望とおわりに

AIは魔法ではありません。
強力なツールを導入したからといって、翌月に全員の業務が変わるなんてことは難易度が鬼なのです。

AI導入で本当に必要なのは、現場の誰かが生み出した「小さな工夫」をこぼさずにすくいあげること。
そして、それを組織の中に伝播させていく仕組みです。

Kodamaが目指すのは、こんな循環が当たり前に回る世界です。

1. 誰かがAIで、面倒な作業の時間を削減する。
2. Agentがそれを静かに検知し、「これ、みんなにも共有していいですか？」と尋ねる。
3. うなずけば、新しい知恵として蓄積される。
4. 翌日、同じ業務で頭を抱えている別の誰かのもとへ、その知見が届く。

今回のハッカソンでは、このループの手前の「手動登録＋セマンティック検索からの提案」までをMVPとして実装しました。
今後はMicrosoft Graphを通じた自動検知や、Power Automate連携まで広げていく予定です。

「孤独なAI活用」を終わらせるため開発した、AI活用浸透エージェント『Kodama』。
誰かの小さな成功体験が、木霊のように、次の誰かのもとへ飛んでいく。
そんな温かいエージェントを作り続けたいと思います。
