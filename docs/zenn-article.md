# はじめに：現場で起きている「孤独なAI活用」

AIの導入が進んでいます。
でも、実際の現場を見ていると、こんな景色が広がっていました。

**数字で見える格差：**
Copilotのログを眺めていると、月に50回以上使い込んでいる人がいる一方、ライセンスを持ちながら一度も使っていない人が同じ部署にいる。

**よく聞こえてきた声：**
- 「触ってみたけど、何に使えばいいか分からなかった」
- 「使いこなしている人が隣にいるらしいけど、話しかけにくい」
- 「DX推進部に聞けばいいのは分かっているけど、毎回それも悪いな…」

活用できている人の成果は確かに出ている。でも、それは孤立した点にすぎなくて、隣の人に伝わっていない。

この「孤独なAI活用」を解消しようと、勉強会資料を作って定期共有する取り組みもやってみました。でも、その場でたくさんの事例を見せたとしても、ピンとこないと意味がない。「これ、今の自分の仕事に関係あるのかな？」と思った瞬間、情報はするっと抜けていく。

もっといい方法があるはずだ、と考えて行き着いたのが**「自発的な事例の流動」**というアイデアでした。

誰かの成功体験が自動的に組織内を徘徊して、必要なタイミングに必要な人のもとへ届く。それも、DX推進部からの一方的なマニュアルとしてではなく、**「同じ会社の○○さんがこれで助かった」という形で。**

そんなエージェントを作りたいという想いで、友人を誘ってMicrosoft Agent Hackathon 2026に臨みました。

# コンセプト：まっさらなAIより、みんなが育てたAIを。

Kodamaが届けたいのは、テキストを整理してくれる便利ツールではありません。

> オフィスの片隅で交わされるこんな会話にレバレッジをかける。
> 「これ、こうやったら半分の時間で終わったんですよ」
> 「この会社でCopilot使うなら、こういう聞き方がコツですよ」

1人が発見した小さな工夫を、次の誰かへ。自然に助け合い、自然に賢くなる。

AIの浸透は、単なる便利ツールの導入ではなく、抽象度の高い業務改革だと思っています。だからこそインパクトが大きい一方、つまずきも大きい。

「仕事を奪われる」という恐怖の問題も、避けて通れません。でも、メール整理・スケジュール調整・資料の初稿作成をAIが担ってくれたとき、残るのは**その会社でその人がやるからこそ面白い仕事**だとも思う。

その一歩を、マニュアルではなく、同僚の体験として届けたい。

そのための道具が、**Kodama**です。

# Kodamaが提供する体験（デモシナリオ）

Kodamaは**「困っている人と成功事例のマッチングを、限りなくシームレスにする」**AI活用提案エージェントです。

一般的なチャットボットとは設計の前提が違います。
チャットボットは「聞かれたら答える」のに対し、Kodamaは「その業務に合うものを、適切なタイミングで、簡単に、具体的に提示する」ことを目指しています。

## 社員視点：今日の成功事例を眺める

Kodamaを開くと、まず**「今日のおすすめ」**が表示されます。

<!-- スクショ: ホーム画面全体（ヘッダー + 今日のおすすめカード + カテゴリグリッド） -->

日付をシードにした決定論的なアルゴリズムで選ばれるため、同じ日にアクセスした全社員が同じ事例を目にします。「あ、さっき見たやつだ」という共通認識が生まれる設計です。

続けて画面に並ぶのは**業務カテゴリのグリッド**。「月次レポート作成」「議事録要約」「提案書作成」といったカテゴリから1つ選ぶと、そのカテゴリに紐づく成功事例がカード形式で並びます。

<!-- スクショ: カテゴリを選択した状態（カテゴリグリッドでいずれかがハイライトされ、事例カードが表示されている） -->

各カードには：
- 誰の事例か（「営業部 佐藤さん」）
- 何が変わったか（「月8h → 2.5h（約70%削減）」）
- 何をしたのか（やり方の説明）
- **今すぐ使えるプロンプト**（コピーボタンで1クリック）

<!-- スクショ: 事例カード拡大（プロンプト枠・「コピーして試す」ボタン・「役立ちましたか？」フィードバックボタンが見える状態） -->

事例カードの下には「役立ちましたか？」というフィードバックボタン（👍 / 🤔）があります。いいね数のランキングは意図的に置きませんでした。「試してみた」という行動履歴はブラウザのlocalStorageに保存され、「最近試したもの」セクションとして自分だけに表示される。**監視されている感覚を与えずに、使用を記録する**という設計です。

---

ここで「なぜチャットUIにしなかったのか」について触れておきます。

Kodamaの最初のプロトタイプはChainlit製のチャットUIでした。でも使ってみると、「何を聞けばいいか分からない」という問題にすぐぶつかりました。AIに詳しくない人ほど、チャットに何を入力すればいいかが分からない。Kodamaが助けたい層は、まさにそこです。

だから、選択肢を先に見せるダッシュボード型にしました。業務カテゴリを選んで、事例を眺めて、気に入ったプロンプトをコピーする。入力は一切不要。

## DX推進担当者視点：誰に何を届けるか

管理者モードでは、困っている社員を選んで「推薦事例 + 戦略の選択肢」が一覧で表示されます。

<!-- スクショ: 管理者モード（左サイドバーにメンバー一覧、右エリアに推薦事例3件が表示されている状態） -->

ユーザーを選ぶと、Agentが**そのユーザーの業務文脈をクエリにしてセマンティック検索**を走らせ、関連度の高い成功事例を最大3件返します。本人の事例は自動で除外されます（「自分の事例を自分に勧める」不自然さを避けるため）。

推薦事例が揃ったら、次の2つの戦略から選べます：

**戦略A：紹介**
「佐藤さまへ。田中さんが月次レポートの効率化で困っており、以前佐藤さまが実施された取り組みが参考になりそうです。15分ほどお時間いただけますか？」
→ メッセージのプレビューをその場で確認できます。

**戦略B：テンプレ共有**
「田中さまへ。月次レポートの効率化に、社内の成功事例（佐藤さん）で使われたプロンプトを共有します。」
→ 成功者が実際に使ったプロンプトが、相手の業務名にローカライズされて生成されます。

<!-- スクショ: 戦略A または B のメッセージプレビューが表示されているモーダルまたはパネル -->

どちらもワンクリックでメッセージプレビューが生成される。「誰に何を届けるか」の判断だけを人間がすればいい設計です。

## 提案は最大3件まで

これはシステム全体で一貫したルールです。AgentのInstructionsにも `top_k` は必ず3以下と明記してあり、APIの `top_k` デフォルト値も3です。

選択肢が多すぎると、人は選べなくなる。Kodamaが提案するのは「どれが最も関連度が高いか」の絞り込みであり、判断疲れを防ぐのも仕事のうちだと考えています。

# 裏側で働く5つのAgentたち（技術構成）

KodamaのバックエンドはAzure AI Foundry Agent Serviceで動く**5つのAgentの協調**によって成り立っています。

## アーキテクチャ全体像

```
[Webダッシュボード (React + Vite + TypeScript)]
         ↕ REST API
[FastAPI バックエンド]
         ↕ Azure AI Foundry SDK
[Orchestrator Agent]
  ├─ Observer Agent   ← Microsoft Graph で観測
  ├─ Collector Agent  ← 困りごとの構造化・本人確認
  ├─ Matcher Agent    ← セマンティック検索・戦略選択
  └─ Proposer Agent   ← 個別提案の生成
         ↕
[Azure OpenAI (GPT-4o 系)]
[Cosmos DB / Azure AI Search]
```

<!-- スクショ: アーキテクチャ図（上記の構成をビジュアル化したもの、またはMermaidレンダリング） -->

## Orchestrator Agent：全体の指揮官

OrchestratorはユーザーとKodamaの**唯一の窓口**です。ユーザーの入力を解釈し、function toolと4つの子Agentを適切に呼び分けます。

Instructionsには判断パターンが明示されています：

- **観測情報の引用**（「田中さんがこう投稿していた」）→ まず`tool_fetch_signals`で補完 → `tool_semantic_search`で関連事例を検索 → 提案
- **能動的な検索依頼**（「高橋さんに合う事例を探して」）→ `exclude_user_id`を設定して`tool_semantic_search`を呼ぶ
- **一般的な質問・雑談** → tool呼び出し不要で通常回答

「最大3件」「戦略A/Bを必ず両方提示」「内部IDを絶対表示しない」「本人の事例を本人に推薦しない」——こうした厳守ルールがInstructionsに直接書かれています。LLMがルールを「覚える」ではなく、**Instructionsで強制する**設計です。

## Observer Agent：組織活動の感覚器

ObserverはMicrosoft Graph APIを通じて、オプトイン済み社員の活動から困りごとのシグナルを検知します。

検知対象は3種類：
1. 「これどうやって」「分からない」といった質問的メッセージ（Teams投稿）
2. Copilotを使い始めて途中で止めた痕跡（セッションログ）
3. 毎月同じ手作業を繰り返しているパターン（Calendarログ）

出力はJSON構造化データで、`source_signal`（`teams_message` / `copilot_abandon` / `repetitive_pattern`）・`raw_excerpt`・`business_context_hint`の3フィールドが含まれます。**Collector Agentへの渡し役**に徹し、永続化は一切行いません。

MVPでは実データ未準備時にモックを返すフォールバックも用意しています。

## Collector Agent：Human-in-the-Loopの守り人

Collectorの仕事は**「データを保存してよいかどうかを確かめること」**です。

Observerから渡された候補シグナルを構造化し、まず本人確認のメッセージを生成します。

> 「月次レポート作業でお困りですか？」

本人が「はい」と回答した場合のみ、`save_pain_point` toolを呼んでCosmos DBに保存します。「困っていない」と回答した場合は永続化せず、同種シグナルを7日間抑制します。

**PII（個人情報）はuser_id（内部ID）のみ扱い、氏名・メールアドレスはDBに保存しません。** 表示時にのみMicrosoft Graphから取得する設計です。

## Matcher Agent：決定論とLLMのハイブリッド

Matcherは「誰と誰を繋ぐか」の判断を担います。設計のポイントは**決定論でスコアを計算し、LLMで最終戦略を選ぶ**ハイブリッド構成です。

スコアリング式：

```python
final_score = (cosine_similarity × 0.7) + (reproducibility_score × 0.3) + business_type_bonus(0.2)
```

- **コサイン類似度（重み0.7）**：困りごとテキストをtext-embedding-3-small（1536次元）でベクトル化し、成功事例との類似度を計算
- **再現可能性スコア（重み0.3）**：その事例が他者でも再現できる可能性（0〜1）。成功者のスキル依存度の逆数として登録
- **業務種別ボーナス（+0.2）**：クエリテキストに成功事例の`business_type`が含まれている場合に加算

embeddingが未登録の場合は`business_type`のテキストマッチへ自動フォールバックします。本番（Azure AI Search）への切り替えも関数単位で差し替え可能な抽象化になっています。

成功事例が0件だった場合（Cold Start）は、業務カテゴリに合うテンプレートプロンプトを提示し、「最初の成功事例を作ってください」と促します。

## Proposer Agent：文脈に合わせた提案の生成

ProposerはMatcherが選んだ成功事例を元に、依頼者向けのメッセージをカスタマイズします。

**戦略A（紹介）の場合：**
```
{成功者名} さま

お忙しい中失礼します。{対象者名} さまが {業務名} の効率化で困っており、
以前 {成功者名} さまが実施された取り組みが参考になりそうです。
15分ほどお時間いただき、ご経験を共有いただけませんでしょうか？
```

**戦略B（テンプレ共有）の場合：**
成功者が使ったプロンプトを、対象者の業務名にローカライズして生成します。「月次レポート作成向けプロンプト」をそのまま渡すのではなく、対象者の業務文脈に合わせた形に書き換えます。

Instructionsには「出典（成功者名）を必ず示す」「押し付けがましい表現を避け、選択肢として提示する」という制約が書かれています。

## 実装上のこだわりポイント

**① フィードバックをサーバーに送らない（MVP段階）**

「役立ちましたか？」ボタンのフィードバックはブラウザのlocalStorageに保存します。サーバーに集計データを送る仕組みはまだ実装していません。これは意図的な判断です——最初から重い集計基盤を作るより、まず使われる体験を作ることを優先しました。Phase 2でCosmos DBのmatch_historyに集約する前提で設計しています。

**② embeddingはlazy loadで、失敗してもサービスを止めない**

FastAPIの起動時（lifespan）に成功事例のembeddingを一括登録します。Azure OpenAIの呼び出しが失敗した場合は、embeddingなしで全件再ロードしてサービスを継続します。検索もembeddingベクトル → テキストマッチへ透過的にフォールバックします。

**③ ConnectedAgentToolで子Agentを「必要なときだけ」呼ぶ**

OrchestratorはConnectedAgentToolで4つの子Agentを保有していますが、「基本はfunction toolで完結させ、判断が難しいときのみ子Agentに委譲する」方針です。LLMコールのコストとレイテンシを抑えながら、複雑なケースへの対応力を維持しています。

**④ DefaultAzureCredentialで認証コードを変えない**

ローカルでは`az login`の認証情報、Azure Container AppsではManaged Identityを自動で使い分けます。環境ごとに認証コードを書き替える必要がなく、シークレットを環境変数に書く必要もありません。

# 今後の展望とおわりに

AIの力だけで魔法のように業務が変わるわけではありません。大切なのは、現場の誰かが生み出した小さな工夫やプロセスを、いかにこぼさず掬い上げ、組織内にやさしく伝播させていくかです。

Kodamaがフルに機能する世界では、こんな流れが自律的に動き続けます。

1. 誰かが月次レポートの時間を半分にした
2. Agentがその変化を検知し、本人に「これ、成功事例として共有していいですか？」と聞く
3. 承認されたら事例DBに追加
4. 翌月、同じ困りごとを持つ別の誰かに届く

このループが回り続けることで、「組織の知」が自然と蓄積されていく。

今回のハッカソンではMVP（手動登録 + セマンティック検索 + 戦略A/B提示）までを実装しましたが、Phase 2以降ではMicrosoft Graphを通じた困りごとの自動検知、戦略選択ロジックの自律更新、Power Automateでの会議自動セットへ段階的に拡張していく予定です。

「AI活用の孤独」を終わらせるための、小さな一歩。
Kodamaという名前には、**木霊**——声が響いて返ってくる——という意味を込めました。誰かの成功体験が、次の誰かのもとに響いて返っていく。そういうエージェントを作り続けたいと思っています。
